# d3-sankey internals — notes for multi-column alluvial

Based on reading `d3-sankey@0.12.3` source at `node_modules/d3-sankey/src/`.
Files: `sankey.js` (main layout), `align.js` (nodeAlign fns), `sankeyLinkHorizontal.js` (path gen), `index.js` (exports).

The package exports exactly: `sankey`, `sankeyLeft`, `sankeyRight`, `sankeyCenter`, `sankeyJustify`, `sankeyLinkHorizontal`. **No `sankeyLinkCircular`** — that lives in the separate `d3-sankey-circular` fork on npm.

---

## 1. nodeAlign — which one for time-series alluvial (depth = year)?

Source (`align.js`):

```js
export function left(node) {
	return node.depth;
}
export function right(node, n) {
	return n - 1 - node.height;
}
export function justify(node, n) {
	return node.sourceLinks.length ? node.depth : n - 1;
}
export function center(node) {
	/* complicated — based on neighbor depths */
}
```

Where `node.depth` = longest incoming path from a source node (computed by BFS in `computeNodeDepths`), and `node.height` = longest outgoing path to a sink.

**For a time-series alluvial with columns = years: use `sankeyLeft`.** It maps every node to `node.depth`, so if year `t` node is reachable only from year `t-1` nodes, it lands in column `t`. `sankeyJustify` (the default) will shove terminal nodes to the right edge — bad if year 2020 node has no outgoing links it jumps to the last column. `sankeyRight` is the mirror problem. `sankeyCenter` centers orphans — also wrong.

Caveat: `computeNodeDepths` uses `depth = longest path from a root`, so an isolated node (no target links) always gets depth=0. If your 2025 column has a "new" node with no history, it will land in column 0 unless you **either** connect it to a prior-year ghost node **or** pass a custom align function.

**Custom align** — simplest way to lock columns to years:

```js
const align = (node) => node.year - startYear; // return the column index directly
sankey.nodeAlign(align);
```

The function receives `(node, n)` and returns the column index. `computeNodeLayers` does `Math.max(0, Math.min(x-1, Math.floor(align(node, x))))`, so you are in full control.

## 2. nodeSort + linkSort — forcing top-to-bottom order

Two separate comparators:

```js
sankey.nodeSort((a, b) => a.rank - b.rank); // within each column
sankey.linkSort((a, b) => a.rank - b.rank); // within each node's incoming/outgoing fan
```

Key source detail from `computeNodeLayers`:

```js
if (sort)
	for (const column of columns) {
		column.sort(sort);
	}
```

And in `relaxLeftToRight` / `relaxRightToLeft`:

```js
if (sort === undefined) column.sort(ascendingBreadth);
```

So if you **set `nodeSort`**, the per-iteration collision pass will NOT re-sort by y0, meaning your order is preserved across all 6 relaxation iterations. If you leave it `undefined`, it re-sorts by current y0 after each relax step (and nodes drift).

**There's also a subtle trap: `nodeSort(null)`** — passing `null` is different from leaving undefined. `null` is truthy for the `if (sort)` check? No — `null` is falsy. So `nodeSort(null)` behaves like unset in `computeNodeLayers`. But in the relax passes, `sort === undefined` is `false` when you passed `null`, so it **skips** the `ascendingBreadth` re-sort. That's the documented idiom for "keep input order, don't auto-sort" — pass `null` explicitly.

```js
sankey.nodeSort(null); // keep insertion order, no auto-y-sort
sankey.linkSort(null); // keep insertion order on links too
```

For strict D/N/S ordering:

```js
const rankMap = { D: 0, N: 1, S: 2 };
sankey.nodeSort((a, b) => rankMap[a.category] - rankMap[b.category]);
sankey.linkSort(
	(a, b) =>
		rankMap[a.source.category] - rankMap[b.source.category] ||
		rankMap[a.target.category] - rankMap[b.target.category],
);
```

## 3. Fixed node positions — can you bypass the layout?

Yes, partially — and there's a **documented escape hatch**: `sankey.update(graph)`.

```js
sankey.update = function (graph) {
	computeLinkBreadths(graph);
	return graph;
};
```

Workflow to fully pre-position nodes:

```js
// 1. Build graph with links manually
const graph = { nodes, links };

// 2. Run computeNodeLinks equivalent — wire sourceLinks/targetLinks yourself
//    OR call sankey(graph) once to let d3-sankey do the wiring,
//    then overwrite y0/y1/x0/x1 on every node.
sankey(graph);

// 3. Overwrite node positions with your manually computed stocks
for (const node of graph.nodes) {
  node.x0 = yearScale(node.year);
  node.x1 = node.x0 + nodeWidth;
  node.y0 = stockScale(node.cumulativeTop);
  node.y1 = stockScale(node.cumulativeTop + node.stockValue);
}

// 4. Set link.width from value directly (since you're not using ky)
const ky = /* your own stock→pixel scale */;
for (const link of graph.links) link.width = link.value * ky;

// 5. Re-run computeLinkBreadths via sankey.update
sankey.update(graph);
```

`sankey.update` only calls `computeLinkBreadths`, which does:

```js
function computeLinkBreadths({ nodes }) {
	for (const node of nodes) {
		let y0 = node.y0;
		let y1 = y0;
		for (const link of node.sourceLinks) {
			link.y0 = y0 + link.width / 2;
			y0 += link.width;
		}
		for (const link of node.targetLinks) {
			link.y1 = y1 + link.width / 2;
			y1 += link.width;
		}
	}
}
```

It stacks outgoing links starting at `node.y0` and incoming links starting at `node.y0` independently. So as long as each node's y0 and each link's `width` are set, `update()` assigns `link.y0`/`link.y1` correctly, and `sankeyLinkHorizontal` will draw the paths.

**Important:** `computeLinkBreadths` stacks sourceLinks from `node.y0` downward, **not** centered on the node. If your node is taller than the sum of outgoing-link widths (intentional gap for "external inflow"), the links will pack against the top edge. To center them, pre-adjust `link.y0`/`link.y1` yourself after `update()` — or even skip `update()` and write `link.y0`/`link.y1` directly.

## 4. sankeyLinkHorizontal with manually positioned nodes

Source:

```js
function horizontalSource(d) {
	return [d.source.x1, d.y0];
}
function horizontalTarget(d) {
	return [d.target.x0, d.y1];
}
export default function () {
	return linkHorizontal().source(horizontalSource).target(horizontalTarget);
}
```

It only reads **four values**: `d.source.x1`, `d.y0`, `d.target.x0`, `d.y1`. Nothing else. You can 100% use it with hand-built link objects:

```js
import { sankeyLinkHorizontal } from "d3-sankey";
const linkPath = sankeyLinkHorizontal();

const myLink = {
	source: { x1: 100 },
	target: { x0: 300 },
	y0: 150, // vertical center of link on source side
	y1: 200, // vertical center of link on target side
	width: 20, // only used if you also draw stroke-width
};

const d = linkPath(myLink); // "M100,150C200,150,200,200,300,200"
```

So for a pure hand-rolled alluvial you can **drop the whole layout** and just import `sankeyLinkHorizontal`. Strokes are drawn via `stroke-width={d.width}` in SVG — d3-sankey never touches SVG, the path is just a centerline Bezier.

## 5. node.value — how incoming vs outgoing mismatch is handled

From `computeNodeValues`:

```js
function computeNodeValues({ nodes }) {
	for (const node of nodes) {
		node.value =
			node.fixedValue === undefined
				? Math.max(sum(node.sourceLinks, value), sum(node.targetLinks, value))
				: node.fixedValue;
	}
}
```

**Answer: it uses `max(incoming, outgoing)`.** And there's the undocumented `node.fixedValue` escape hatch — set it on a node before calling `sankey()` and that node's height maps to `fixedValue` instead of the max.

```js
// Force a node to represent total housing stock,
// regardless of how much flow actually transits through it
node.fixedValue = totalStock;
```

This is the cleanest way to get "a node is bigger than its flows" behavior — the extra space becomes visual padding (or top-of-node gap, depending on layout).

`initializeNodeBreadths` uses a single `ky` across all columns:

```js
const ky = min(columns, (c) => (y1 - y0 - (c.length - 1) * py) / sum(c, value));
```

i.e. the scale is chosen so the tallest column exactly fills the available height. With `fixedValue`, you can pin stocks to a consistent pixel scale across years.

## 6. Multi-column, same categories at each step — the idiom

There is no special API. The canonical pattern:

- One node per `(category, timestep)` → id like `"D-2018"`, `"D-2019"`, ...
- One link per `(category, t) → (category, t+1)` with value = stock at `t` (or some quantity that flows through).
- Use a custom `nodeAlign` that returns the timestep index directly:

```js
const align = (node) => node.timestep;
sankey.nodeAlign(align);
```

- Use `nodeSort` keyed on category rank to lock top-to-bottom order.
- For identity links (same category across time with no change), you still need a link with nonzero value, otherwise the node's `value` collapses to whatever flows through.

**The trick for preserving stock invariants** (inflow must equal outflow for a conserved quantity like housing units): add explicit "inflow from outside" and "outflow to outside" nodes per column, or use `fixedValue` to force node height to the true stock and compute link widths manually.

## 7. Gaps / uncovered portions — showing inflows from outside

Three approaches, from easiest to most flexible:

### (a) Phantom external node per column

Add a "source" node in column `-1` (or hidden off-canvas) that emits links into real nodes. Its height can be clipped. Works with stock d3-sankey.

### (b) `node.fixedValue`

Set `node.fixedValue = trueStock`. The node will be tall enough to represent the real stock. Outgoing links stack from the top (per `computeLinkBreadths`), leaving uncovered space at the bottom. Post-process to re-center if needed.

### (c) Fully manual — compute all positions, use only `sankeyLinkHorizontal`

Skip `sankey()` entirely. You compute `x0/x1/y0/y1` per node using your own stock scale, then for each link compute `link.y0` and `link.y1` as the vertical center on the source and target sides (honoring stock positions), and feed each link object to `sankeyLinkHorizontal()`. This is what you want for a precise multi-column stock-flow alluvial where gaps represent "external inflow".

Rough sketch:

```js
import { sankeyLinkHorizontal } from "d3-sankey";

// Inputs: years, categories, stockData[year][category], flowData[year][cat→cat]
const linkGen = sankeyLinkHorizontal();

// 1. Pre-compute node positions from stocks
const nodes = [];
for (const year of years) {
	let yCursor = 0;
	for (const cat of categories) {
		const h = stockScale(stockData[year][cat]);
		nodes.push({
			id: `${cat}-${year}`,
			year,
			cat,
			x0: xScale(year),
			x1: xScale(year) + nodeWidth,
			y0: yCursor,
			y1: yCursor + h,
		});
		yCursor += h + nodePadding;
	}
}
const nodeById = new Map(nodes.map((n) => [n.id, n]));

// 2. Pre-compute link endpoints — track cursor on each side
const sourceCursor = new Map(); // node.id -> next y offset (outgoing)
const targetCursor = new Map(); // node.id -> next y offset (incoming)
const links = [];
for (const year of years.slice(0, -1)) {
	for (const [from, to, value] of flowData[year]) {
		const s = nodeById.get(`${from}-${year}`);
		const t = nodeById.get(`${to}-${year + 1}`);
		const w = flowScale(value);
		const sOff = sourceCursor.get(s.id) ?? 0;
		const tOff = targetCursor.get(t.id) ?? 0;
		links.push({
			source: s,
			target: t,
			width: w,
			y0: s.y0 + sOff + w / 2, // centerline of stroke on source side
			y1: t.y0 + tOff + w / 2, // centerline on target side
		});
		sourceCursor.set(s.id, sOff + w);
		targetCursor.set(t.id, tOff + w);
	}
}

// 3. Draw
links.forEach((l) => {
	svg.append("path")
		.attr("d", linkGen(l))
		.attr("fill", "none")
		.attr("stroke", colorFor(l))
		.attr("stroke-width", l.width)
		.attr("opacity", 0.5);
});
```

The gap between `sourceCursor.get(s.id)` and the full node height `s.y1 - s.y0` is visually "uncovered" — that's your external inflow space.

## 8. Source-code flow of `sankey()`

Main entry (`sankey.js` line 66):

```js
function sankey() {
	const graph = { nodes: nodes.apply(null, arguments), links: links.apply(null, arguments) };
	computeNodeLinks(graph); // wire sourceLinks/targetLinks, resolve id→node refs
	computeNodeValues(graph); // node.value = fixedValue ?? max(sum in, sum out)
	computeNodeDepths(graph); // BFS from roots → node.depth (longest path from source)
	computeNodeHeights(graph); // BFS from leaves → node.height (longest path to sink)
	computeNodeBreadths(graph); // x0/x1 via align, y0/y1 via iterative relaxation
	computeLinkBreadths(graph); // stack link.y0/y1 from node.y0
	return graph;
}
```

### computeNodeLinks

Wires `source`/`target` string ids to object refs, populates `node.sourceLinks` and `node.targetLinks` arrays, assigns `node.index` and `link.index`. If `linkSort` is set, sorts each node's link arrays once here.

### computeNodeValues

Sets `node.value` to `max(sum(sourceLinks), sum(targetLinks))` OR `node.fixedValue` if defined. This is the only place `fixedValue` is consulted — it's a node property, not a sankey option.

### computeNodeDepths / computeNodeHeights

BFS-based. `depth` = longest path from any source node (no target links). `height` = longest path to any sink. Throws `"circular link"` if BFS doesn't terminate within `nodes.length` iterations. **Isolated nodes get depth=0 and height=0.**

### computeNodeBreadths (the big one)

1. `computeNodeLayers` — uses `align(node, x)` to bucket nodes into columns, assigns `x0`/`x1` linearly based on extent and `nodeWidth`. If `nodeSort` is set, sorts each column once.
2. `initializeNodeBreadths` — chooses a single scale `ky` = minimum over columns of `(height - padding) / sum(values)`. Then for each column, packs nodes top-to-bottom at `y = node.value * ky`, then re-centers by distributing leftover space proportionally across gaps.
3. Iterate `iterations` (default 6) times: `relaxRightToLeft` then `relaxLeftToRight`. Each pass moves a node toward the weighted barycenter of its link neighbors, scaled by `alpha = 0.99^i`. After each column is relaxed, `resolveCollisions` pushes overlapping nodes apart starting from the middle.
4. **Key insight:** if `nodeSort` is set, the per-iteration `column.sort(ascendingBreadth)` is skipped, so your order is preserved even as positions jitter.

### computeLinkBreadths

Stacks outgoing links downward from `node.y0` (each link center at `y0 + width/2`, then `y0 += width`). Same for incoming links, independently. **No centering** — outflows pack against top of node.

### Customization surface

- `sankey.nodeId(fn)` — custom id accessor for link source/target resolution
- `sankey.nodeAlign(fn)` — returns column index `(node, n) → int`
- `sankey.nodeSort(fn | null)` — column ordering; pass `null` to lock insertion order
- `sankey.nodeWidth(px)` — default 24
- `sankey.nodePadding(px)` — default 8 (vertical gap between nodes in same column)
- `sankey.nodes(fn | array)` — data accessor
- `sankey.links(fn | array)` — data accessor
- `sankey.linkSort(fn | null)` — link ordering in fans
- `sankey.size([w, h])` or `sankey.extent([[x0,y0],[x1,y1]])` — canvas bounds
- `sankey.iterations(n)` — relaxation iterations, default 6
- **Per-node: `node.fixedValue`** — overrides computed value
- **Per-node post-layout: `node.x0/x1/y0/y1`** can be overwritten and `sankey.update(graph)` re-runs only `computeLinkBreadths`

What's NOT customizable without forking:

- The relaxation algorithm itself
- The collision-resolution strategy
- Link stacking direction (always top-down from `node.y0`)
- The single global `ky` scale (no per-column scaling)

## 9. Alternatives if d3-sankey is too constraining

1. **Use only `sankeyLinkHorizontal`** — skip `sankey()` entirely, hand-compute positions, draw with the path generator. Shown in section 7(c) above. Probably the right move for a stock-flow alluvial where you already know the node heights from data.

2. **Use `d3-shape` directly** — `sankeyLinkHorizontal` is just:

    ```js
    linkHorizontal()
    	.source((d) => [d.source.x1, d.y0])
    	.target((d) => [d.target.x0, d.y1]);
    ```

    You can customize this further, e.g. swap `linkHorizontal` for `linkVertical` or write a custom Bezier with different control-point placement.

3. **`d3-sankey-circular` fork** — npm package that adds `sankeyCircular()` layout and `sankeyLinkCircular` path generator for graphs with backward links. Not needed for time-forward alluvials but worth knowing.

4. **Custom Bezier** — for full control of curve shape:
    ```js
    function linkPath(d) {
    	const x0 = d.source.x1,
    		x1 = d.target.x0;
    	const xi = d3.interpolateNumber(x0, x1);
    	const x2 = xi(0.5),
    		x3 = xi(0.5);
    	return `M${x0},${d.y0}C${x2},${d.y0} ${x3},${d.y1} ${x1},${d.y1}`;
    }
    ```
    This is literally what `linkHorizontal()` generates internally.

## TL;DR for this project

For a time-series alluvial with columns = years and conserved stocks (housing units) where you want to show inflows from outside:

- **Skip `sankey()` layout** — you already know stock heights per year per category.
- **Compute node positions manually** using a single vertical stock scale shared across all columns.
- **Use `sankeyLinkHorizontal()`** as the path generator only. It needs `source.x1`, `target.x0`, `d.y0`, `d.y1` per link — nothing else.
- **Track `sourceCursor` / `targetCursor` per node** to stack links, leaving uncovered node space where external inflows/outflows should be.
- **Render link strokes** with `stroke-width={link.width}` in SVG where `width = flowValue * ky`.

This sidesteps every constraint of `d3-sankey`'s layout and gives pixel-perfect stock alignment across columns. The 374-line existing `d3-alluvial-chart.tsx` in this folder likely already takes this or a similar approach — check it before building from scratch.
