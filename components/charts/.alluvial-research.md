# D3 Alluvial Chart Research — Housing Fate 2019-2025

Target: `components/charts/d3-alluvial-chart.tsx`. The component must render a **multi-column** alluvial (not a single Sankey flow, not a stacked area), with 7 yearly columns, 3 categories each, and ribbons showing persistence + late transitions.

---

## a) Best references (in priority order)

1. **Elijah Meeks — "Alluvial Charts and Their Discontents"** (Medium).
   Opinionated piece from the author of _D3.js in Action_. Key lesson for us: alluvial charts aggregate bigrams, so **readers cannot trace an individual "path" through >2 columns** — the visualization is honest only if we treat each year-to-year ribbon as an independent flow. Do NOT imply that a single unit of housing has an identity across all 7 years. Interactivity (hover highlighting a single stage) is how serious alluvials earn trust.
   https://medium.com/@Elijah_Meeks/alluvial-charts-and-their-discontents-10a77d55216b

2. **Joe Bottigliero — "Sankey Diagram to Alluvial Diagram"** (Observable).
   The canonical trick to get a multi-column alluvial out of `d3-sankey`: **duplicate each conceptual node as `id.source` + `id.target`** so every link is perfectly linear. For our use case (7 columns, fixed category order) we don't even need this, but the pattern of _pre-linearising_ links is the right mental model.
   https://observablehq.com/@jbottigliero/sankey-diagram-to-alluvial-diagram

3. **Fil — "Sankey link paths (#91)"** (Observable).
   Offset-path technique for rendering ribbons as **filled polygons** rather than stroked beziers. Fixes the "shearing" artifact when a ribbon changes width mid-curve. Relevant if we ever need tapered ribbons (inflow → outflow widths differ).
   https://observablehq.com/@fil/sankey-link-paths

4. **Ian Johnson (enjalot) — "Weird Sankey Links"** (Observable).
   Custom `sankeyLinkPath(link)` that builds a **closed filled polygon** with two bezier edges. Best reference for the ribbon path primitive we should use instead of `sankeyLinkHorizontal` if nodes get crowded.
   https://observablehq.com/@enjalot/weird-sankey-links

5. **Jason Davies — Parallel Sets** (`d3-parsets`).
   Historically the "best-in-class" parallel-categories vis in D3. Older but the visual idioms are still canonical: bars-with-width, sorted by frequency, hover-to-highlight a single ribbon end-to-end.
   https://www.jasondavies.com/parallel-sets/ and https://github.com/jasondavies/d3-parsets

6. **Rosvall & Bergstrom 2010 — "Mapping Change in Large Networks"** (PLOS ONE).
   The _original_ alluvial paper — this is where the idiom was invented. Two design rules we should steal directly: (a) **block heights = volume**, (b) **order blocks within a column by size (largest on top)** so the eye reads the dominant category first. Our case is different (we have a forced category order for storytelling), but the "big on top" default is the neutral fallback.
   https://journals.plos.org/plosone/article?id=10.1371/journal.pone.0008694

## c) Design idioms we should adopt

### Ordering within columns

- **Rule 1 (forced order for storytelling):** when categories have a narrative meaning (Sold / Non-profit / Developer-held), keep a fixed vertical order across all columns. This is what makes persistence read as a straight horizontal band. Current code does this — keep it.
- **Rule 2 (link ordering to minimize crossings):** for each source node, outgoing links must be **sorted so that the link going to the topmost target is stacked first** (i.e., at the top of the source block). Symmetrically for incoming at each target. This is exactly the Rosvall rule and it is what keeps the "persistence" band straight while pushing transitions to the edges. The current code does this correctly.
- **Rule 3 (transition links on top):** when persistence and transition links coexist at the same source, stack the persistence portion flush against its own category band and push transitions to the _outside_ edge (top if going up, bottom if going down). This makes late-sales ribbons visible as diagonal accents instead of swallowing the middle.

### Ribbon rendering: stroke-with-width vs filled polygon

- **Stroke-width + `sankeyLinkHorizontal`** is fine _if_ the ribbon has the same width at both ends. Ours does (persistence = same value at both ends), so this works for the common case.
- **Filled polygon** is required if source-width ≠ target-width (e.g. tapered ribbons for partial flows). Use Ian Johnson's pattern:
    ```js
    function sankeyLinkPath(link) {
    	const sx = link.source.x1,
    		tx = link.target.x0;
    	const sy0 = link.y0 - link.width / 2,
    		sy1 = link.y0 + link.width / 2;
    	const ty0 = link.y1 - link.width / 2,
    		ty1 = link.y1 + link.width / 2;
    	const mx = (sx + tx) / 2;
    	return `M${sx},${sy0} C${mx},${sy0} ${mx},${ty0} ${tx},${ty0}
              L${tx},${ty1} C${mx},${ty1} ${mx},${sy1} ${sx},${sy1} Z`;
    }
    ```
    Render with `fill` instead of `stroke` — no more shearing.

### Opacity / color conventions

- **Persistence ribbons:** fill with the _source_ category color, opacity **0.35–0.45**. They should read as a solid band.
- **Transition ribbons:** fill with the _target_ category color (where the unit _ends up_, not where it came from), opacity **0.55–0.65**, so they pop against the persistence band.
- **Hover:** bump the hovered ribbon to opacity 0.9, desaturate all others to 0.12. This is the standard Parallel Sets idiom and the one Elijah Meeks argues is non-negotiable for alluvials.
- Current code uses stroke opacity 0.32 / 0.55 — directionally right. Consider bumping transition to 0.6 so late-sales pop harder.

### New inflow

Three idioms, pick one:

1. **External downward arrow above the column** (what current code does). Honest: shows mass entering the system from outside. Good.
2. **Uncovered top portion of the node block** — leave the top `inflow * yScale` of the node unfilled or hatched. Harder to read but visually integrated.
3. **Phantom "inflow" source node** in a zero-width column. Cleanest semantically but adds chart complexity.
   **Keep the arrow approach** — it matches the storytelling ("~4000 new units land each year, and here's where they end up") and doesn't compress the main blocks.

### Labels

- **Year labels below columns** (current code) is correct for time-series alluvials. Above is reserved for category headers in non-temporal parallel sets.
- **Block labels:** show only when a block is tall enough (e.g. `heightPx > 14`). Put the label _next to_ the block (right side for leftmost column, left side for rightmost, inside for interior) — never on top of ribbons.
- **Numeric callouts:** show absolute value on hover; don't clutter the base chart with permanent numbers.

### Scale compression

When one category dominates (our "Sold" category will likely be 70-80% of every column), the small categories become unreadable. Options:

- **Square-root y-scale:** `yScale = sqrt(value) * k`. Visually understates the dominant class, overstates the minor ones. Honest-ish if the story is about structure, not magnitude.
- **Log-transform:** too aggressive here; reserve for 3+ orders of magnitude.
- **Stay linear but add a "min-height floor"** (e.g. `Math.max(heightPx, 6)`) so small categories never disappear. Mark the floor with a subtle visual cue (tick, hatch) so readers know it's been clamped.
- **Small-multiples sidebar:** a separate tiny chart next to the main one, showing the minor categories zoomed. Best honesty/readability tradeoff but more layout work.

**Recommendation for our case:** linear scale + min-height floor of 4px, with a footnote noting the floor.

### Hover/tooltip (desktop web)

- Hover a **ribbon** → highlight it end-to-end, dim everything else, show `"YYYY → YYYY: N units (category A → category B)"`.
- Hover a **node block** → highlight all incoming and outgoing ribbons, show `"YYYY · Category: N units (X% of stock)"`.
- Do NOT trigger tooltips on the svg root or on invisible padding — scope listeners to the actual rect / path elements.
- Use React state + className-based dimming (no d3 selection.transition) to stay idiomatic with the rest of `blog/components/charts/`.

## d) Specific code patterns worth copying

### Pattern 1: Manual `d3-sankey` with fixed order (preferred alternative base)

```js
import { sankey, sankeyLinkHorizontal, sankeyLeft } from "d3-sankey";

const layout = sankey()
	.nodeId((d) => d.id)
	.nodeWidth(12)
	.nodePadding(6)
	.nodeAlign(sankeyLeft) // respects explicit layer
	.nodeSort(null) // null = preserve input order
	.linkSort(null) // null = preserve input order
	.extent([
		[0, 0],
		[innerWidth, innerHeight],
	]);

// Pre-assign layer/depth so d3-sankey doesn't re-compute them:
nodes.forEach((n) => {
	n.layer = n.column;
});
const graph = layout({ nodes, links });
// Now graph.nodes have correct x0/x1/y0/y1; render rects + sankeyLinkHorizontal paths.
```

This is worth trying as a replacement for the hand-rolled offset math in the current file — it's ~30 lines shorter and handles edge cases d3-sankey has been debugging for a decade.

### Pattern 2: Filled-polygon ribbon (Ian Johnson) — for tapered flows

See snippet above under "Ribbon rendering". Use when `link.width_source !== link.width_target`.

### Pattern 3: Ribbon offset stacking (Rosvall rule)

Our current code already implements this correctly:

```js
// For each source, sort outgoing by target.y0; assign cumulative offset.
// For each target, sort incoming by source.y0; assign cumulative offset.
```

This is the correct crossing-minimization heuristic for a fixed-order alluvial. Keep it — it's textbook.

### Pattern 4: Gradient fills on links (Bottigliero)

Ribbons can fade from source color to target color with an SVG `linearGradient` per link. Visually lovely for transitions, but **overkill and slow for 7 columns × 9 ribbons = ~60 gradients**. Use solid fill keyed on target color instead (cheaper, still readable).

### Pattern 5: Hover dim-all-but-one

```jsx
const [hoveredId, setHoveredId] = useState(null);
// on each path: opacity={hoveredId && hoveredId !== link.id ? 0.08 : baseOpacity}
```

Standard React idiom, no d3-selection required.

## Summary for our chart

**Keep the current manual-layout architecture.** Apply these changes:

1. Implement node + ribbon hover with dim-others pattern.
2. Bump transition ribbon opacity 0.55 → 0.6.
3. Add a `Math.max(heightPx, 4)` floor for tiny categories.
4. Add block labels conditional on `heightPx > 14`.
5. Consider swapping the hand-rolled layout for `d3-sankey({nodeSort:null, linkSort:null})` — fewer lines, battle-tested.
6. Prepare (but don't yet use) the Ian Johnson filled-polygon link generator for the day we need tapered ribbons.

Animation is not critical. If added later, the standard idiom is to enter ribbons by interpolating `strokeDashoffset` from the full path length down to 0 (gives a "flowing water" effect) — but this is cosmetic and not where the value is.
